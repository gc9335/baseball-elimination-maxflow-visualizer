# 实验六：棒球淘汰问题设计说明

## 1. 目标

使用 Python 完成棒球淘汰问题实验，交付可运行代码、程序说明和 Markdown 实验报告。程序需要：

1. 实现 Edmonds–Karp 基准最大流算法。
2. 实现带分层图和当前弧优化的 Dinic 算法。
3. 将棒球淘汰问题转换为最大流问题。
4. 输出每支球队是否被淘汰及其证明集合。
5. 测试题目中的四支球队和 Princeton 官方测试数据。
6. 比较两种算法的正确性与运行时间，并生成性能对比图。

## 2. 技术方案

### 2.1 最大流算法

基准算法采用 Edmonds–Karp。每轮通过 BFS 在残量网络中寻找最短增广路，直到不存在增广路。其时间复杂度为：

```text
O(VE²)
```

优化算法采用 Dinic：

1. BFS 构造分层图。
2. DFS 在分层图中发送阻塞流。
3. 使用当前弧数组跳过已经证明无法继续增广的边。

一般网络上的理论复杂度为：

```text
O(V²E)
```

两种算法共用统一的残量边数据结构和求解接口，以保证棒球淘汰模块可以无差别切换算法，也便于进行公平的性能比较。

### 2.2 棒球淘汰网络

判断目标球队 `x` 是否仍有机会并列第一。令：

```text
M = wins[x] + remaining[x]
```

即假设目标球队赢下自己的所有剩余比赛后能够达到的最大胜场数。

首先进行直接淘汰检查：若存在球队 `i` 满足：

```text
wins[i] > M
```

则目标球队已经被直接淘汰，无需构造流网络。

否则建立如下网络：

- 源点 `s`
- 每一对不包含目标球队的球队 `(i, j)` 对应一个比赛节点
- 每支非目标球队对应一个球队节点
- 汇点 `t`

边和容量如下：

1. `s -> game(i,j)`，容量为 `games[i][j]`，表示这两队之间尚未分配的胜场。
2. `game(i,j) -> team(i)`，容量为无穷大。
3. `game(i,j) -> team(j)`，容量为无穷大。
4. `team(i) -> t`，容量为 `M - wins[i]`，限制球队 `i` 最终不能超过目标球队的最大胜场数。

记源点所有比赛边的容量和为：

```text
G = Σ games[i][j], i < j, i != x, j != x
```

若最大流等于 `G`，所有剩余比赛都能在不使其他球队超过 `M` 的条件下分配，因此目标球队仍未被淘汰。若最大流小于 `G`，则不存在可行的胜负分配，目标球队被淘汰。

最大流结束后，在残量网络中从源点仍可达的球队节点构成最小割证明集合，可用于解释哪些球队共同导致目标球队被淘汰。直接淘汰时，证明集合使用胜场数已经超过 `M` 的球队。

## 4. 输入与输出

### 4.1 输入格式

兼容 Princeton Baseball Elimination 数据格式：

```text
球队数量
球队名 已胜 已负 剩余 与各球队的剩余比赛数...
```

例如：

```text
4
Atlanta       83 71 8  0 1 6 1
Philadelphia  80 79 3  1 0 0 2
New_York      78 78 6  6 0 0 0
Montreal      77 82 3  1 2 0 0
```

解析器将检查：

- 球队数和数据行数一致。
- 每行字段数正确。
- 数值非负。
- 赛程矩阵对称。
- 对角线为零。
- 每支球队的剩余场数不小于矩阵行和；差值允许表示与分区外球队的比赛。

### 4.2 命令行

默认使用 Dinic：

```powershell
python main.py data/teams4.txt
```

指定算法：

```powershell
python main.py data/teams4.txt --algorithm edmonds-karp
python main.py data/teams4.txt --algorithm dinic
```

批量测试与绘图：

```powershell
python scripts/benchmark.py
```

单支球队结果包含：

- 是否淘汰。
- 直接淘汰或非直接淘汰。
- 最大可能胜场数。
- 最大流值与待分配比赛总量。
- 淘汰证明球队集合。

## 5. 测试设计

### 5.1 最大流单元测试

使用具有已知最大流值的小型网络验证：

- 空网络或源汇不连通。
- 单路径网络。
- 多条并行增广路径。
- 需要使用反向残量边修正先前选择的网络。
- Edmonds–Karp 与 Dinic 输出一致。

### 5.2 棒球淘汰测试

`teams4.txt` 的预期结论：

- Atlanta：未淘汰。
- Philadelphia：被淘汰，证明集合为 Atlanta 和 New_York。
- New_York：未淘汰。
- Montreal：被 Atlanta 直接淘汰。

同时对 Princeton 官方提供的数据逐一运行两种算法，检查：

- 每支球队的淘汰布尔值一致。
- 最大流值一致。
- 证明集合满足最小割性质。
- 已知示例结果与官方说明一致。

### 5.3 性能实验

性能测试分为两部分：

1. 官方数据集：记录每个文件、每种算法完成全部球队判定的总耗时。
2. 可复现随机数据：使用固定随机种子生成不同球队规模的合法对称赛程，覆盖更大的网络规模。

为了减少计时噪声：

- 每个样本预热一次。
- 每种算法重复运行若干次。
- 记录中位数耗时。
- 两种算法使用完全相同的输入。
- 性能测试前先验证结果一致。

输出 CSV 字段至少包括：

```text
dataset, teams, algorithm, repeat, elapsed_ms
```

性能图使用分组柱状图：

- 横轴：数据集或球队规模。
- 纵轴：中位运行时间（毫秒，对跨度较大时使用对数刻度）。
- 系列：Edmonds–Karp、Dinic。
- 图中不使用会误导读者的三维效果。

## 7. 错误处理与可复现性

- 文件不存在、格式错误或赛程数据不一致时，输出明确错误信息并返回非零退出码。
- 下载失败不会破坏已有数据；脚本会指出未能取得的文件。
- 随机测试固定种子。
- 图表和 CSV 均由脚本重新生成，不依赖手工录入。
- Python 代码使用类型标注，并避免依赖第三方最大流库。
- 第三方依赖仅用于测试和绘图，核心算法全部自行实现。
